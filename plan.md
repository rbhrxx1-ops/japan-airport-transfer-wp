# Japan Airport Transfer 举牌迎送服务 WordPress 网站实施计划

> **执行状态（2026-07-15 GMT+9）：** 固定页头生产发布、招聘页草稿上传、完整备份、桌面/移动端独立验收与临时代码清理已完成。生产主题样式版本为 `0.2.5`，招聘页 ID `94` 保持 `draft`。详细证据见 `docs/production-deployment-2026-07-15.md`。

## 一、计划目标

本计划用于在 **不直接破坏生产站、全过程可追踪、每阶段可验收与可回滚** 的前提下，完成 `meetandgreet.jp` 的专属 WordPress 网站与在线预约程序。主域名于 2026-07-15 从 `japanairporttransfer.com` 迁移为 `meetandgreet.jp`。网站前端、系统提示、法务页面及客户邮件全部使用自然日语，不设置语言切换器。

首期将交付四项标准服务：`空港お迎え（ミート）`、`空港お見送り（センディング）`、`駅お迎え`、`駅お見送り`。`乗継サポート` 与 `アテンドサービス` 因 PDF 正文尚未完成，只建立日语咨询入口，不开发未经确认的标准流程、价格或自动预约逻辑。

> **提交语义：** 在线提交只代表 `受付済み`，不代表 `予約確定`。价格、资源、集合点与服务可用性由客服人工确认后才能形成正式预约。

## 三、已确认的产品与技术基线

| 维度 | 决策 |
|---|---|
| 网站形态 | 专属 WordPress 主题 `jat-meet-theme` + 专属预约插件 `jat-reservation` |
| 编辑方式 | 原生区块编辑器、受控区块与结构化字段，不依赖重型页面构建器 |
| 前端语言 | 100% 日语；仅品牌名、URL、机场代码、航班号及客户输入可保留必要外文 |
| 核心地点 | 羽田空港、成田空港、東京駅、品川駅；其他地点为咨询入口 |
| 预约流程 | 五步向导：服务与日期 → 旅客信息 → 迎送内容 → 申请人/车辆 → 确认与同意 |
| 订单存储 | 专属结构化数据表、版本化迁移、公开受付编号、状态历史、邮件日志与审计 |
| 状态机 | `受付済み → 確認中 → お見積り送付済み → 予約確定 → サービス完了`，并支持 `変更確認中`、`キャンセル` |

| 参考站使用 | 只借鉴栏目组织、信任表达和用户路径，不复制文字、图片、地图、商标、代码或视觉 |
| PDF使用 | 作为业务流程依据；其中带日期的航站楼、车寄、集合点与时限必须重新核验后才能公开 |

### 阶段 M3：五步在线预约前端与提交接口

预约程序使用轻量原生 JavaScript 或最小组件实现，不引入不必要的前端框架。表单支持条件字段、返回编辑、错误恢复、移动端键盘、欢迎牌文字预览和最终摘要。

| 步骤 | 日语界面 | P0字段与逻辑 | 验收重点 |
|---:|---|---|---|
| 1 | サービス・ご利用日時 | 服务、地点、日期时间、航班/列车、起讫信息 | 机场/车站与迎接/送行分支正确 |
| 2 | ご利用者情報 | 姓名、人数、行李、紧急电话、行动支持 | 国际电话、数字边界、长度校验 |
| 3 | お出迎え・お見送り内容 | 欢迎牌、语言、附加协助、完成报告 | 欢迎牌安全转义；只显示真实可提供选项 |
| 4 | お申込者・車両情報 | 个人/法人、公司、联系人、目的地、司机/车辆 | 条件字段切换后不遗留旧数据 |
| 5 | 入力内容の確認 | 分区摘要、修改入口、隐私/条款/最终同意 | 提交文案明确为申请受理 |

提交接口实施服务器端白名单、类型/长度校验、输出转义、预处理查询、Nonce、蜜罐/反垃圾、速率限制、幂等键和重复提交检测。网络错误、Nonce 过期或弱网重试不得清空已填内容，也不得生成重复订单。

**M3检查点：** 四项 P0 服务各完成正常提交与关键错误路径；两个咨询分支只采集需求；成功页返回不可顺序推测的受付编号；所有状态和错误均为自然日语。

### 阶段 M4：订单后台、状态机、邮件、权限与隐私

预约业务存储在插件专属表中，避免把全部敏感订单散落在文章元数据。数据库迁移必须版本化、可重复执行，并为后台筛选建立必要索引。

| 模块 | 实现内容 | 验收标准 |
|---|---|---|
| 数据层 | 订单、行程、状态历史、邮件日志；文件表仅在未来开启上传时建立 | 迁移可重复执行，数据不随主题切换丢失 |
| 后台列表 | 受付编号、服务、地点、日期、申请人、公司、状态、提交日期筛选 | 默认遮蔽电话、邮箱和详细行程 |
| 订单详情 | 原始快照、当前资料、同意记录、备注、状态历史、邮件记录 | 修改者、时间与前后值可追踪 |
| 状态机 | 确定性允许转换，不合理跳转拒绝 | `キャンセル` 不能直接变为 `サービス完了` |
| 权限 | 客服、主管、管理员能力分离 | 普通编辑者不可导出或查看全部个人资料 |
| CSV | 按筛选结果即时生成并记录审计 | 不长期保存在公开媒体目录 |
| 邮件 | 受付、报价、确定、变更、取消、完成、失败告警；HTML+纯文本 | 主题、正文、按钮、页脚均为日语，失败可追踪 |
| 隐私 | 同意版本/时间、保留期限、导出与删除支持 | 日志、上传、备份遵守同一保留策略 |

**M4检查点：** 从申请到完成/取消的业务闭环通过；每次状态变化可审计；越权测试、邮件测试与 CSV 审计通过。

SEO 以高质量服务页与地点页为主，不批量生成重复机场页面。每个页面有唯一日语标题、描述、H1、规范链接和可见 FAQ；只有页面真实显示的 FAQ 才输出结构化数据。

**M5检查点：** 形成性能报告、无障碍报告、日语字符串审计与内容核验记录；阻断项清零后进入综合验收。

| 测试组 | 关键用例 | 通过条件 |

## 九、主要风险与处理策略

| 风险 | 影响 | 处理策略 |
|---|---|---|
| 当前 WordPress 环境未知 | 插件冲突、版本不兼容、生产中断 | M0 只读审计、完整备份、独立测试环境、禁止直接生产试错 |
| 目标域名公开索引状态不明确 | 上线后 SEO 或重复内容问题 | 审计现有索引、canonical、robots、站点地图与旧 URL，制定重定向表 |
| PDF 现场信息可能过期 | 客户到错集合点或服务承诺错误 | 每个地点保存核验日期和责任人；业务确认后发布 |
| 资料含未确认时限与能力 | 形成错误合同承诺 | 未确认内容进入内部清单，不进入公开页面或自动逻辑 |
| 个人资料和行程敏感 | 泄露、越权、保留过久 | 最小采集、权限分离、遮蔽、审计、保留期限、备份治理 |
| 邮件送达不稳定 | 客户误以为申请失败或客服漏单 | 使用可靠SMTP/事务邮件、发送日志、失败告警和重试机制 |
| 日语默认字符串遗漏 | 品牌体验不一致、用户困惑 | 单一字符串治理、自动扫描与人工全路径审计 |
| 竞品与PDF版权 | 侵权风险 | 仅借鉴结构；使用原创文案、授权素材和自制可维护示意图 |
| 生产与 Git 不一致 | 无法回滚与追踪 | 所有发布从版本化代码产生；上线后回归并核对最终 hash |
